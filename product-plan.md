# "就它了"App 产品战略规划

## 📋 文档概览
- **项目名称**: 就它了 (PickIt)
- **核心定位**: 解决外卖点餐选择困难症的智能决策助手
- **商业模式**: 美团/饿了么API嫁接，通过返佣链接获得佣金
- **目标用户**: 有选择困难症的外卖用户（学生、上班族、年轻人）
- **开发状态**: 需求明确，环境就绪，明日开始编码
- **规划时间**: 2026-03-27（产品战略规划）

---

## 1. 项目概述与愿景

### 1.1 项目背景
在美团、饿了么等外卖平台选择泛滥的时代，用户面临"选择过载"困境。面对数百家餐厅、上千道菜品，决策时间过长，体验下降，甚至产生焦虑。"就它了"App应运而生，将"更多选择"转换为"唯一选择"。

### 1.2 愿景使命
- **愿景**: 成为外卖决策的第一入口
- **使命**: 用智能推荐消除选择焦虑，简化点餐流程
- **口号**: "选择困难？就它了！"

### 1.3 核心价值主张
1. **对用户**: 3秒完成点餐决策，节省时间，减少焦虑
2. **对平台**: 增加订单转化率，提升用户体验
3. **对我们**: 通过返佣创造可持续收入

## 2. 市场分析与机会

### 2.1 市场现状
- **外卖市场规模**: 2025年中国外卖市场规模预计超1.5万亿元
- **用户痛点**: 67%的用户表示"选择太多不知道吃什么"
- **竞争格局**: 美团、饿了么主导，但专注于"决策辅助"的垂直应用稀缺

### 2.2 目标市场规模
- **潜在用户**: 中国外卖用户约4亿，30%有选择困难 ≈ 1.2亿人
- **初期目标**: 获取100万用户，1%市场份额
- **用户价值**: 单个用户年均点餐100次，客单价40元，年交易额4000元

### 2.3 市场机会点
1. **差异化定位**: 不做外卖平台，做"决策引擎"
2. **轻资产模式**: 无需自建配送、供应链
3. **用户粘性**: 解决高频痛点，建立使用习惯
4. **数据价值**: 积累用户饮食偏好数据

## 3. 用户画像与痛点

### 3.1 核心用户画像
| 维度 | 学生用户 | 上班族用户 |
|------|----------|------------|
| **年龄** | 18-25岁 | 22-35岁 |
| **场景** | 宿舍、图书馆 | 办公室、家中 |
| **痛点** | 预算有限、时间紧张 | 工作忙、决策疲劳 |
| **频次** | 每周3-5次 | 每周4-6次 |
| **客单价** | 15-25元 | 30-50元 |

### 3.2 用户痛点深度分析
1. **选择过载**: 平台提供过多选择，反而难以决策
2. **决策疲劳**: 每天重复选择，消耗心理能量
3. **时间浪费**: 平均每次点餐耗时15-20分钟
4. **后悔情绪**: 选完后觉得"也许另一个更好"
5. **社交压力**: 为他人点餐时承担决策责任

### 3.3 用户旅程地图
```
发现痛点 → 下载App → 注册引导 → 首次推荐 → 下单体验 → 重复使用 → 社交分享
```

## 4. 核心解决方案

### 4.1 解决方案框架
**核心逻辑**: 个性化推荐 + 游戏化决策 + 一键下单

### 4.2 两种模式设计（已有）
#### 4.2.1 轻度选择困难症模式
- **推荐数量**: 3家店的菜
- **交互**: "就它了" vs "换一个"按钮
- **场景**: 日常点餐，有一定选择余地

#### 4.2.2 重度选择困难症模式  
- **推荐数量**: 1家店的菜
- **交互**: "接受挑战"按钮
- **场景**: 极度纠结，希望彻底摆脱选择
- **积分激励**: 1元消费=1积分，兑换虚拟奖品

### 4.3 推荐算法设计
#### 初期规则引擎
1. **过滤层**: 22菜系偏好 + 17忌口（绝对过滤）
2. **排序层**: 
   - 历史好评率（权重30%）
   - 距离远近（权重25%）
   - 价格预算（权重20%）
   - 等待时间（权重15%）
   - 随机惊喜（权重10%）

#### 中期机器学习
- 基于用户反馈（点击、下单、评价）优化权重
- 协同过滤（相似用户偏好）
- 时间序列分析（用餐时间、季节偏好）

## 5. 商业模式：API嫁接与返佣

### 5.1 收入来源
**核心**: 美团/饿了么联盟营销（CPS模式）

#### 5.1.1 返佣机制
- **美团联盟**: 平均返佣率 1-3%（根据品类）
- **饿了么星选**: 平均返佣率 1.5-3.5%
- **淘宝客**（外卖）：返佣率 0.5-2%

#### 5.1.2 预估收入模型
| 指标 | 保守 | 乐观 |
|------|------|------|
| 日活跃用户 | 10,000 | 50,000 |
| 下单转化率 | 15% | 25% |
| 日均订单量 | 1,500 | 12,500 |
| 平均客单价 | 35元 | 45元 |
| 平均返佣率 | 2% | 2.5% |
| **日均收入** | **1,050元** | **14,063元** |
| **月收入** | **31,500元** | **421,875元** |
| **年收入** | **378,000元** | **5,062,500元** |

### 5.2 成本结构
1. **开发成本**: 初期自主开发，无人力成本
2. **服务器成本**: 初期月均500元（用户量<10万）
3. **营销成本**: 用户获取成本估算5-10元/人
4. **维护成本**: 月均2,000元（后期）

### 5.3 盈利路径
**阶段1** (0-3月): 验证模式，获取1万用户
**阶段2** (4-6月): 优化算法，达到10万用户，月收入3万+
**阶段3** (7-12月): 规模化，50万用户，月收入20万+
**阶段4** (1年+): 探索多元化收入（广告、数据服务、订阅制）

### 5.4 API接入策略
#### 技术实现路径
1. **初期**: 使用公开的联盟链接跳转

## 7. 用户留存策略（交互设计核心）

### 7.1 留存驱动设计
#### 7.1.1 首次体验优化
- **极简注册**: 3分钟内完成偏好设置
- **即时满足**: 注册后立即获得推荐
- **成就反馈**: "恭喜完成个性化设置！"

#### 7.1.2 日常使用激励
1. **积分系统**
   - 1元消费 = 1积分
   - 积分兑换: 外卖红包、视频会员、实物奖品
   - 等级体系: 青铜→白银→黄金（增加特权）

2. **游戏化元素**
   - 连续使用打卡（7天奖励）
   - 解锁成就（"川菜达人"、"健康饮食者"）
   - 随机彩蛋（特殊日子推荐特色菜）

3. **社交粘性**
   - 分享今日推荐（社交炫耀）
   - 为朋友点餐功能（解决社交决策压力）
   - 饮食偏好对比（"你和朋友的口味相似度"）

#### 7.1.3 防止流失机制
1. **智能提醒**: 根据历史用餐时间推送推荐
2. **个性化挽留**: 长期未使用用户收到专属推荐
3. **反馈循环**: 推荐后询问满意度，持续优化

### 7.2 交互界面设计原则
#### 7.2.1 "药丸式"按钮设计（已有概念）
- **视觉**: 圆角大按钮，友好触感
- **交互**: 点击有明显反馈（颜色变化、轻微动画）
- **状态**: 选中/未选中状态清晰区分

#### 7.2.2 两步走界面流程

#### 7.2.3 推荐页面设计
- **突出主推菜品**: 大图+吸引人描述
- **简洁决策按钮**: "就它了"(主) vs "换一个"(次)
- **进度暗示**: "今日已为您节省15分钟决策时间"

## 8. 技术架构

### 8.1 技术栈选择
- **前端**: React Native（Android优先，iOS后期）
- **后端**: Node.js + Express/NestJS
- **数据库**: PostgreSQL（主库）+ Redis（缓存）
- **部署**: 云服务器（阿里云/AWS）

### 8.2 架构设计
```
用户端(React Native) ←→ API网关 ←→ 后端服务 ←→ 数据库
                          ↓
                   外卖平台API（美团/饿了么）
```

### 8.3 关键技术点
1. **推荐引擎**: 初期规则引擎，后期迁移到机器学习
2. **用户画像存储**: PostgreSQL JSON字段存储复杂偏好
3. **实时位置**: 获取用户位置优化距离计算
4. **数据同步**: 与外卖平台订单数据定期同步对账

### 8.4 数据模型（核心）
```sql
-- 用户表
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  preferences JSONB,  -- {cuisines: [...], avoidances: [...], budget: ...}
  created_at TIMESTAMP
);

-- 推荐记录表
CREATE TABLE recommendations (
  id SERIAL PRIMARY KEY,
  user_id INTEGER,
  restaurant_id VARCHAR,
  dish_id VARCHAR,
  displayed_at TIMESTAMP,
  clicked BOOLEAN,
  ordered BOOLEAN
);

-- 订单跟踪表
CREATE TABLE orders (
  id SERIAL PRIMARY KEY,
  user_id INTEGER,
  platform VARCHAR,  -- 'meituan' or 'eleme'
  order_id VARCHAR,
  amount DECIMAL,
  commission DECIMAL,
  created_at TIMESTAMP
);
```
